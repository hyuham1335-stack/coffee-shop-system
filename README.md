## 1. 프로젝트 개요

## 4. 설계 의도

### 도메인 분리 및 결합도 최소화
- Order는 과거 주문 데이터이고, UserPoint는 현재 상태 데이터로 역할이 다르다고 판단
- 두 도메인을 @ManyToOne 을 사용하여 직접 연관시키지 않고 userId 기반 참조만 사용하여 결합도 최소화


### 사용자 & 포인트 테이블 통합
- 이름, 이메일 등 별도의 사용자 정보를 다루지 않는 요구사항
- 사용자 식별값만 존재하기 때문에 User + Point를 분리하지 않고 UserPoint 단일 테이블로 구성


### 이벤트 기반 구조 (Kafka 대체)
- 토이 프로젝트에서는 인프라 복잡도를 고려하여 @EventListener 기반으로 주문/결제 후처리를 수행
- 추후 프로젝트 규모가 커질 경우 Kafka를 사용하여 대체

## 5. 기술적 선택 이유

### 5.1 포인트 충전 - 낙관적 락 사용

#### ✅선택 이유
- 포인트 충전 트랜잭션이 짧고 단순함
- 동일 사용자의 동시 충전 가능성이 낮음
- 충돌 발생 확률이 낮다고 판단

#### ✅ 비교
- 비관적 락
  - 충돌이 적고 재시도 로직이 단순하여 굳이 DB 커넥션을 점유하는 비관적 락을 사용할 이유가 없다고 판단
 
- 분산 락
  - 트래픽 규모가 작으며 충돌 가능성이 낮아 Redis 외부 서버에 의존성을 늘리면서 분산 락을 사용할 이유가 적다고 판단
 

### 5.2 주문 / 결제 - 비관적 락 사용

#### ✅선택 이유
- 하나의 요청에 여러 작업 포합
  - 포인트 잔액 확인
  - 포인트 차감
  - 주문 생성
  - 결제 생성
- 비관적 락을 통해 여러 작업을 원자적 작업으로 묶고 정합성을 보장

#### ✅ 비교
- 낙관적 락
  - 하나의 요청에 여러 작업이 묶여있기 때문에 낙관적 락을 사용할 경우 재시도가 필요한 경우 로직이 복잡해질 수 있음

- 분산 락
  - 트래픽의 규모가 작아서 Redis 외부 인프라에 의존하고 결제 로직의 복잡도를 높히기 보다는 비관적 락을 사용하는 것이 적합하다고 판단
  - 트래픽의 규모가 증가하거나 한정 메뉴 등의 추가로 주문 / 결제 트래픽이 몰릴 경우에 분산 락을 통해 DB 병목을 줄이는 것이 적합하다고 판단
